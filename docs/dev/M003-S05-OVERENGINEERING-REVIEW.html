M003 / S05 · Architecture review

Slice Lifecycle Overengineering Review

Historical snapshot: this review describes the pre-S05 baseline at f7dbea8524. S05 has since implemented the Slice lifecycle cutover. See the architecture overview and lifecycle command integration runbook for current contracts.

Conclusion: the subsystem is not suffering from too many sophisticated abstractions. Its main cost is the opposite: four slice verbs are spread across parallel legacy and canonical write paths, transport adapters, and projection conventions. The best refactor is a narrow convergence onto the existing Domain Operation and lifecycle primitives, followed by deletion of obsolete adapters and renderers.

Findings

High priorityHigh change riskFragmented authority

1. The four verbs do not share one durable lifecycle command boundary

Why this is avoidable complexity: the code has one accepted authority model and already has the reusable machinery to implement it, yet equivalent slice decisions are encoded differently depending on the verb. This forces every maintainer to reason about legacy hierarchy status, canonical lifecycle status, Domain Operation provenance, projection work, and transport replay separately.

Evidence

  • tools/replan-slice.ts:117-324 uses executePlanningDomainOperation, adopts/transitions canonical lifecycles, records a durable event and Projection Work, and returns committed/replayed status.
  • db/writers/cascades.ts:140-195, :53-85, and :208-243 implement complete, reopen, and skip as direct legacy-table transactions without a Domain Operation or canonical slice transition.
  • tools/reassess-roadmap.ts:406-444 maps a removed slice to legacy skipped plus canonical cancelled. Direct gsd_skip_slice only executes skipSliceCascade, so two user routes for removing work produce different canonical state.
  • tools/complete-slice.ts:377-470 commits legacy completion before summary/UAT content and gate results. :495-563 persists gates and compatibility events later, outside that transaction.
  • docs/dev/architecture.md:36 explicitly records that file-derived slice completion, UAT, and broader lifecycle routing have not fully cut over.

Impact

  • Retry behavior is verb-specific: replan has a durable idempotent receipt; complete/reopen/skip do not.
  • Canonical shadow and legacy status can disagree after skip, complete, or reopen.
  • Callers must remember which side effects are authoritative and which are best-effort for each verb.
  • Fault tests need to cover combinations of crash windows rather than one transaction contract.

Smallest safe simplification

Reuse the existing Domain Operation kernel through a narrow slice-lifecycle adapter. Inside each command's mutate callback, call context-bound leaves extracted from the transaction-owning cascades, then transition the slice/task lifecycle rows. Record the verb's immutable event and durable Projection Work in the same operation. Keep one explicit writer per verb—completeSlice, reopenSlice, cancelSlice, and the existing replanSlice—rather than building a generic command registry.

Blast radius / sequencing

High. This crosses database revision fencing, public tools, legacy status compatibility, projection work, and restart behavior. Land one verb at a time behind contract tests. Start with skip/cancel because the semantic mismatch is observable and the handler is smallest; complete should land last because it also owns gates, UAT, and closeout content.

Medium priorityMedium change riskDuplicate adapters

2. Pi and workflow MCP maintain parallel slice adapters that can drift

Why this is overengineered: the transport layer has both a shared executor module and locally reimplemented adapters. The extra paths do not protect meaningful transport variability; they repeat database-open checks, error shaping, handler invocation, state rebuilds, and response text.

Evidence

  • tools/workflow-tool-executors.ts:948-987 exports executeSliceReopen. Workflow MCP calls it at packages/mcp-server/src/workflow-tools.ts:1205-1213.
  • Pi ignores that executor and reimplements reopen at bootstrap/db-tools.ts:1440-1475.
  • The workflow contract names executeSkipSlice at packages/contracts/src/workflow.ts:108-113, but no such executor exists. Pi implements skip at bootstrap/db-tools.ts:988-1054; workflow MCP implements another version at packages/mcp-server/src/workflow-tools.ts:2633-2654.
  • The variants return different text, detail fields, and error semantics. Executor failures set isError: true; some local registration failures do not.

Impact

Behavior can differ by transport even when both routes ultimately call the same core handler. A change to state refresh, worktree routing, error codes, or a new invocation identity must be made in several places and is easy to test on only one surface.

Smallest safe simplification

Add the contract-promised executeSkipSlice. Route Pi reopen and skip through loadWorkflowExecutors(), as complete and replan already do. Keep only transport-owned work outside the executor: schema parsing, base-path selection, write-gate enforcement, serialized invocation, and private invocation identity.

Blast radius

Medium. Response text and isError behavior may be observable. Pin one parity table across direct executor, Pi registration, and workflow MCP before deleting either adapter.

Medium priorityHigh change riskCommand / projection coupling

3. Completing a slice is also a 25-field Markdown authoring protocol

Why this is overengineered: one lifecycle command accepts identity, narrative, UAT prose, requirement changes, dependency metadata, file summaries, observability notes, and gate evidence. Transport coercion and renderer recovery exist largely because the lifecycle mutation is coupled to a document-shaped payload.

Evidence

  • types.ts:750-803 defines 25 complete-slice fields, most of which exist to populate sections or frontmatter.
  • workflow-tool-executors.ts:1046-1114 spends roughly 70 lines coercing scalar/array and string/object variants before calling the handler.
  • tools/complete-slice.ts:104-289 hand-renders summary/UAT Markdown and contains a parser for three requirement sections.
  • On duplicate recovery, tools/complete-slice.ts:426-455 reads stored rendered Markdown and parses sections back into completion input. That makes a projection-shaped blob participate in reconstructing a command.
  • ADR-046 requires every machine-relevant fact to have a normalized representation and forbids reconstructing workflow truth from narrative prose (docs/dev/ADR-046-database-authoritative-workflow-lifecycle.md:74-104).

Impact

  • Natural tool use is harder because the agent must fill document metadata while making a lifecycle decision.
  • Formatting changes affect transport schemas and completion behavior.
  • Projection recovery depends on heading spellings and prose separators.
  • Tests protect coercion and template plumbing instead of primarily protecting closeout authority.

Smallest safe simplification

Do not introduce another generic DTO. Persist normalized closeout facts and evidence in the Domain Operation, retain at most one narrative note as canonical DB content, and render SUMMARY/UAT from those rows. During compatibility, accept the current fields in the adapter and translate them once; keep them out of the core complete-slice writer.

Blast radius / sequencing

High because this changes a public MCP schema and DB representation. First move the existing payload into normalized DB writes without changing the external schema. Only then deprecate redundant input fields. Do not combine this with the initial lifecycle cutover PR.

Low priorityLow change riskDuplicate work

4. Complete-slice renders the milestone roadmap twice

Evidence: tools/complete-slice.ts:473-489 calls renderRoadmapFromDb and verifies the returned content. Later, :542-545 calls renderMilestoneShellProjections, whose first step is another renderRoadmapFromDb call at workflow-projections.ts:450-455.

Impact: every successful completion repeats a database read and disk write, and each path has its own failure logging. The duplication also makes it less obvious which render attempt establishes projection freshness.

Recommendation: perform one shell projection pass and verify the resulting roadmap file/content once. Keep the summary/UAT write separate until durable Projection Work owns those artifacts. Avoid adding a configurable projection pipeline just to remove this duplicate call.

Risk: Low. Preserve the existing “DB completion remains committed” result and stale-projection reporting in focused failure tests.

Medium priorityMedium change riskObsolete abstraction

5. A known-lossy PLAN renderer remains exported and heavily tested after production stopped using it

Evidence: workflow-projections.ts:54-119 contains renderPlanContent and renderPlanProjection. renderAllProjections explicitly refuses to call them because they omit important sections and corrupt multi-line task descriptions (:485-489). Missing-file recovery uses the authoritative renderPlanFromDb instead (:577-582). Production searches find no caller of renderPlanProjection; direct uses are tests. Several test suites continue to protect the simplified renderer.

Impact: maintainers see two apparent PLAN rendering authorities, and tests increase confidence in a path the runtime intentionally avoids. Future work can accidentally resurrect the lossy renderer.

Recommendation: remove renderPlanProjection and renderPlanContent after confirming they are not part of a supported package export. Delete or redirect their abstraction-focused tests to markdown-renderer.ts behavior. Keep the regression test that proves renderAllProjections preserves an authoritative PLAN.

Risk: Medium, solely because an exported function may have unknown out-of-repo consumers. If this module is private to the bundled extension, risk drops to Low.

Optional cleanupLow change riskPremature generality

6. Complete-slice has a generic gate-dispatch loop for one actual gate

Evidence: tools/complete-slice.ts:57-67 maps gate IDs through a switch that only handles Q8. The handler then builds a registry map and loops pending gates at :500-525. Adding a gate is not actually a one-place change: the registry and this switch must both change.

Impact: small, but it makes Q8 closure harder to follow and suggests extension capacity the implementation does not currently have.

Recommendation: while moving completion into a Domain Operation, express Q8 closure directly as one typed closeout rule. Generalize only when a second complete-slice-owned gate has materially different field mapping.

Risk: Low. Preserve pending-to-pass/omitted semantics and unknown-gate behavior.

What should stay

Transaction-safe cascade semantics

db/writers/cascades.ts puts structural guards and related hierarchy writes in one transaction and returns discriminated outcomes. This prevents TOCTOU bugs and is good behavior to preserve. Extract context-bound leaves for Domain Operations; do not nest the transaction-owning cascade or inline its policy back into handlers.

Canonical lifecycle primitives

adoptLifecycleIfMissing and adoptOrTransitionLifecycle support three real hierarchy kinds and dozens of callers. Their identity validation, provenance checks, state versions, and transition policy justify the abstraction.

Projection failure isolation

Lifecycle completion surviving a projection failure is required behavior. The current handlers consistently treat projection writes as repairable follow-on work. Preserve that property while moving the retry record into durable Projection Work.

Layout-aware cleanup

Reopen cleanup handles both legacy and flat layouts because both layouts exist in deployed projects. That branching is compatibility complexity, not speculative flexibility. Consolidate it only behind an already-needed projection ownership API.

Projection flush seam

projection-flush.ts is only 20 lines, but it has 11 production callers and gives mutation exits one name for a real responsibility. It should eventually delegate to the durable projection worker; deleting the seam now would scatter calls.

Replan preservation rules

Replan deliberately retains removed task identities, blocks edits to completed work, and transitions omitted work to cancelled. Those are user-visible integrity rules, not ceremony. Move them intact if extracting a writer.

Recommended implementation order

  1. Pin cross-transport parity. Add one table-driven contract covering Pi and workflow MCP outcomes for success, missing DB, stale/replayed invocation, blocked write gate, and projection failure.
  2. Unify skip/cancel first. Decide its canonical meaning, add executeSkipSlice, record a Domain Operation plus canonical cancelled lifecycle, and route both transports through it.
  3. Unify reopen. Give reopen invocation identity, transition the slice and resettable task lifecycle heads inside the Domain Operation, and replace the Pi-local adapter with executeSliceReopen.
  4. Unify complete. Add evidence/closeout preconditions and idempotent operation identity before changing its public content schema. Keep projections noncritical.
  5. Shrink projection code. Remove the second roadmap render and delete the lossy PLAN renderer/tests after the supported-export check.
  6. Normalize closeout content last. Translate the existing 25-field transport payload into normalized DB facts, then deprecate projection-only fields over a compatibility window.

Checkpoint rule: each verb should land independently with restart, exact replay, stale revision, canonical/legacy shadow, projection obstruction, and transport parity tests green before the next verb moves.

Open questions

  1. Does “skip” mean permanent cancellation, temporary deferral, or either based on reason? Reassess currently treats removal as canonical cancellation. The direct skip tool needs one explicit semantic before cutover; do not average the two.
  2. Which closeout fields are machine-relevant? Requirements, verification, UAT mode/verdict, dependency outputs, and gate results need normalized rows. Narrative, lessons, and prose summaries may remain canonical DB content rendered verbatim.
  3. Is workflow-projections.ts a supported external API? If not, delete the unused PLAN exports directly. If yes, deprecate them with a clear replacement rather than silently preserving an unsafe renderer.
  4. What is the compatibility window for the complete-slice tool schema? Transport adapters can absorb old fields while the core writer becomes narrow, but removal should be versioned and tested.
  5. Should reopen invalidate downstream evidence/decisions now or in a later slice? ADR-046 requires dependency-reachable revalidation. That behavior should be named and tested, not smuggled into generic projection cleanup.
