M003 S06 Milestone Lifecycle Overengineering Review

Scope: Current Milestone complete, validate, reopen, recovery, settlement, projection, transport, and compatibility paths. Reviewed handlers, database writers and Domain Operations, callers, tests, architecture/runbook documentation, adapters, and projection decisions.

Conclusion: the subsystem is overengineered mainly because the same lifecycle decision is implemented in several partially overlapping paths. The code already has the right deep primitive—revision-checked Domain Operations—but Milestone lifecycle still bypasses it. S06 should consolidate authority, not add another coordinator.

Recommended order: establish one Milestone Domain Operation, route every mutation through it, make closeout readiness a database-only query, then reduce transport and projection duplication. Keep git/worktree effects and rendering as post-commit adapters.

Problem and Current Scale

The user need is small: validate a Milestone, complete or reopen it once, preserve history, automate repair, and show readable status. The current implementation distributes that behavior across three public handlers, at least six direct completion writers/replayers, three closeout proof/settlement surfaces, filesystem cleanup and mirroring, and two Milestone-reopen transport adapters.

Findings

High Milestone completion has several independent writers

Category: Distributed authority and duplicated policy.

Evidence: the normal handler directly calls updateMilestoneStatus(..., "complete") in tools/complete-milestone.ts:149-203. Merge finalization does the same in worktree-lifecycle.ts:96-109; startup journal recovery in auto-start.ts:233-261; stuck recovery in auto-recovery.ts:218-257; and JSONL replay in workflow-reconcile.ts:196-208. These paths enforce different prerequisites. The runbook explicitly says Milestone commands still lack the revision/Authority-Epoch fence and durable receipt at docs/dev/lifecycle-command-integration-runbook.md:94-98.

Impact: retries can duplicate effects, stale recovery can win over newer work, and each new invariant must be copied into every writer. This is the largest source of drift and test burden.

Smallest safe simplification: add one milestone-lifecycle-domain-operation.ts beside the proven Slice operation. Give it complete and reopen entry points; put legacy hierarchy mutation, canonical lifecycle transition, evidence checks, event/outbox, Projection Work, and receipt in the same transaction. Convert merge, startup, recovery, reconciliation, Pi, and MCP paths to requests into that operation. Do not introduce a generic manager or strategy registry.

Change risk: High—many callers and recovery paths change, so land characterization and replay/fault tests first.

High Projection files still create or decide workflow truth

Category: Authority inversion.

Evidence: closeout-consistency-gate.ts:75-153 treats Slice SUMMARY files as evidence and can manufacture a passing Milestone assessment plus four quality gates. milestone-readiness.ts:38-48 classifies a Milestone terminal when hasSummary is true. milestone-validation-verdict.ts:44-77 falls back from the database to three file locations. milestone-closeout.ts:224-245 parses VALIDATION and requires Slice SUMMARY files during dispatch. This conflicts with ADR-046’s one-way projection rule at docs/dev/ADR-046-database-authoritative-workflow-lifecycle.md:74-112.

Impact: a stale or copied Markdown file can authorize closeout, while a missing projection can block valid database work. The repair code then needs more repair code.

Smallest safe simplification: remove file fallback from normal readiness, validation, and completion decisions. Query normalized assessment, verdict, UAT, Waiver, child lifecycle, and evidence rows only. Missing files enqueue/repair Projection Work and set a stale diagnostic; they never create a pass or block a valid mutation. Preserve file ingestion only in the explicit import flow.

Change risk: Medium—legacy projects may depend on implicit recovery, so provide one explicit import/repair command and a clear diagnostic.

Medium Closeout policy is split across overlapping gate, proof, coordinator, and settlement layers

Category: Layering without distinct ownership.

Evidence: milestone-closeout.ts is 300 lines and repeats validation, UAT, child-summary, implementation, and operational checks. It then calls checkCloseoutConsistencyGate at milestone-closeout.ts:214-221. milestone-closeout-proof.ts:97-123 wraps that gate with optional artifact and implementation checks. milestone-settlement.ts:35-80 calls the proof again to choose a pause message. Option flags such as allowOpenMilestone, allowPassThroughValidation, refreshFromDisk, and two implementation-evidence modes make the same concept mean different things by caller.

Impact: maintainers must navigate four modules to answer “may this Milestone complete?”, and tests protect combinations of compatibility options rather than one business invariant.

Smallest safe simplification: create one pure database closeout-readiness query returning typed blockers. Let the complete Domain Operation revalidate the same predicate transactionally. Keep one host settlement orchestrator for git/worktree effects and one independent projection-health check. Delete wrapper options as their callers migrate.

Change risk: Medium—message formatting and host-effect sequencing are external behavior; keep typed blocker codes stable during consolidation.

Medium Lifecycle handlers mix mutation, rendering, reconciliation, and telemetry

Category: Too many responsibilities.

Evidence: tools/complete-milestone.ts:130-278 validates, transacts, renders YAML/Markdown, chooses a path, avoids overwrite, clears three caches, flushes projections, writes a manifest, appends JSONL, logs partial failures, and shapes the public receipt. tools/reopen-milestone.ts:59-137 performs the cascade, manually deletes Milestone/Slice/UAT/Task files for legacy and flat layouts, flushes, writes a manifest, and logs an event.

Impact: replay behavior and projection currentness are hard to prove. Every file-layout change touches lifecycle code, and every lifecycle change risks projection behavior.

Smallest safe simplification: make handlers thin: validate transport input, invoke the Domain Operation, then ask one projection dispatcher to render or remove artifacts only if the operation still owns the current lifecycle head. Manifest and compatibility JSONL should be projection destinations, not independent best-effort hooks in handlers.

Change risk: Medium—projection filenames are compatibility contracts, but the move can preserve exact output.

Medium One aggregate validation verdict is expanded into four apparently independent gates

Category: False precision.

Evidence: milestone-validation-gates.ts:18-51 documents that individual gate verdicts are unavailable, then writes every MV gate with the same overall verdict and empty findings. tools/validate-milestone.ts:177-197 stores the aggregate assessment and these cloned gates in one transaction. Other closeout code later reasons about pending/closed gates as if they were independent evidence.

Impact: extra rows and APIs suggest a proof granularity the input does not support; a single weak overall pass becomes four passes.

Smallest safe simplification: either store only the aggregate Milestone verdict, or change validation input to provide a verdict and evidence reference for each required criterion. Prefer the first for S06 unless per-criterion evidence already exists canonically.

Change risk: Medium—consumers of MV01–MV04 need a compatibility view or migration.

Low Reopen has two transport adapters with the same behavior

Category: Redundant forwarding layer.

Evidence: tools/workflow-tool-executors.ts:1113-1151 exports executeMilestoneReopen, but bootstrap/db-tools.ts:1421-1483 independently opens the database, imports the handler, maps errors, and formats a nearly identical result. Search finds no production caller for the shared executor.

Impact: public details and error semantics can diverge when S06 adds replay, duplicate, superseded, and stale fields.

Smallest safe simplification: register the tool through executeMilestoneReopen, as complete and validate already do, then delete the direct adapter.

Change risk: Low—add one executor-contract test before deletion.

Medium Park/unpark remains a second, file-first Milestone lifecycle

Category: Compatibility behavior posing as authority.

Evidence: milestone-actions.ts:57-97 writes PARKED.md first and treats database update failure as a warning; milestone-actions.ts:106-132 similarly removes the marker before best-effort DB sync; milestone-actions.ts:190-209 answers parked state and reason only from the file. The S05 runbook explicitly defers park/unpark until after S06 at lifecycle-command-integration-runbook.md:94-98.

Impact: folding this into S06 would expand blast radius, but ignoring it indefinitely preserves a second Milestone authority.

Smallest safe simplification: keep park/unpark out of the S06 implementation, record it as the next bounded lifecycle cutover, and stop adding new consumers of PARKED.md. Later, make the database mutation authoritative and render PARKED.md as a projection.

Change risk: Medium—guided flow and queue selection consume this state.

What Should Stay

  • The deep Domain Operation primitive is justified. db/domain-operation.ts:21-72 provides revision, Authority Epoch, idempotency, event, outbox, and Projection Work semantics used across real transports and concurrent workers.
  • Slice lifecycle is the right implementation pattern. slice-lifecycle-domain-operation.ts:80-102 builds a fenced request; complete/reopen/cancel each commit one semantic event and projection target and return replay-aware receipts.
  • Git/worktree publication is a real boundary. Host effects can fail independently and need durable settlement, but they should return typed results rather than mutate lifecycle directly.
  • Legacy/flat path handling is real compatibility. Centralize it in projection renderers instead of deleting it during lifecycle convergence.
  • Focused tests are valuable. Current tests cover closeout, projection staleness, merge recovery, and live state-machine failures; reuse them as characterization before deleting compatibility paths.

Open Questions

  1. Does S06 complete the Milestone before or after required git/worktree merge receipts? ADR-046 says required host effects settle before lifecycle completion; make that ordering explicit in one acceptance test.
  2. Is Milestone validation one aggregate verdict or a bundle of criterion verdicts? The schema and API should represent one model, not both.
  3. Which legacy projects still rely on implicit VALIDATION/SUMMARY backfill? Measure this before removing fallback, then route those cases through explicit import.
  4. Should reopening a Milestone preserve completed child work, or intentionally reset the entire hierarchy? Current behavior resets everything; confirm this product rule before encoding it canonically.
  5. When will park/unpark converge? Give it an explicit owner so S06 does not absorb it accidentally.
